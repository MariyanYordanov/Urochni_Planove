#### **Час 12: Cryptographic Hash Functions (40 мин)**

**Теоретична част (24 мин):**

**🔢 Hash Function Properties и Applications (10 мин)**

**Fundamental Hash Function Properties:**

**Deterministic:**
- **Consistent output** - same input always produces same hash
- **Reproducibility** - enables verification и validation
- **Implementation independence** - different implementations produce identical results
- **Critical за integrity** - allows detection на data modifications

**Fixed Output Size:**
- **Uniform length** - regardless на input size, hash length constant
- **Space efficiency** - large files represented by small hash values
- **Storage benefits** - easier to store и transmit hash values
- **Indexing applications** - consistent size enables efficient data structures

**Avalanche Effect:**
- **Small input changes** - single bit change affects approximately half output bits
- **Unpredictability** - minor modifications produce drastically different hashes
- **Security implication** - prevents prediction на hash values
- **Quality metric** - good hash functions exhibit strong avalanche effect

**Cryptographic Properties:**

**Pre-image Resistance (One-way):**
- **Computational infeasibility** - given hash H, cannot find message M such що H = hash(M)
- **Forward computation easy** - calculate hash от message quickly
- **Reverse computation hard** - determine message от hash computationally infeasible
- **Security foundation** - enables password hashing и commitment schemes

**Second Pre-image Resistance:**
- **Given message M1** - computationally infeasible to find different M2 where hash(M1) = hash(M2)
- **Targeted collision resistance** - protection против specific message attacks
- **Integrity guarantee** - ensures specific message cannot be replaced
- **Digital forensics** - maintains evidence integrity

**Collision Resistance:**
- **No two messages** - computationally infeasible to find any M1 ≠ M2 where hash(M1) = hash(M2)
- **Strongest property** - implies both pre-image resistances
- **Digital signature security** - prevents signature forgery attacks
- **Birthday paradox** - limits effective security level to n/2 bits за n-bit hash

**📊 Hash Function Families и Evolution (8 мин)**

**MD (Message Digest) Family:**

**MD5 (1991):**
- **Output size:** 128 bits (32 hexadecimal characters)
- **Security status:** Cryptographically broken, collision attacks practical
- **Historical importance:** Widely used в 1990s и early 2000s
- **Current usage:** Legacy systems only, не recommended за security applications
- **Vulnerabilities:** Collision attacks found в 2004, further weaknesses discovered

**SHA-1 (Secure Hash Algorithm 1):**
- **Output size:** 160 bits (40 hexadecimal characters)
- **Security status:** Deprecated, theoretical collision attacks demonstrated
- **Timeline:** NIST standard 1995-2017, phased out в major applications
- **Notable attacks:** SHAttered attack (2017) demonstrated practical collisions
- **Legacy support:** Still used в Git, но migrating to SHA-256

**SHA-2 Family:**

**SHA-256:**
- **Output size:** 256 bits (64 hexadecimal characters)  
- **Security level:** 128-bit equivalent strength
- **Current status:** Widely adopted, considered secure
- **Applications:** Bitcoin, TLS, digital signatures, certificate authorities
- **Performance:** Good software performance, hardware acceleration available

**SHA-512:**
- **Output size:** 512 bits (128 hexadecimal characters)
- **Security level:** 256-bit equivalent strength
- **64-bit optimization:** Designed за efficient execution на 64-bit systems
- **Applications:** High-security applications, long-term archival
- **Variants:** SHA-512/256, SHA-512/224 за truncated outputs

**SHA-3 (Keccak):**
- **Different construction** - sponge function instead на Merkle-Damgård
- **Security margin** - large security margins, resistant to length extension
- **NIST standard** - approved в 2015 като alternative to SHA-2
- **Performance** - slower в software, efficient в hardware
- **Future-proofing** - backup solution если SHA-2 vulnerabilities discovered

**🛠️ Hash Function Applications (6 мин)**

**Password Storage:**
- **Never store plaintext** - passwords hashed before database storage
- **Salt usage** - random value added to prevent rainbow table attacks
- **Key derivation functions** - PBKDF2, Argon2, scrypt за password hashing
- **Iteration counts** - multiple rounds to increase computational cost
- **Verification process** - compare hash на entered password със stored hash

**Digital Forensics:**
- **File integrity** - prove files unchanged since collection
- **Evidence chains** - cryptographic proof на evidence handling
- **Data deduplication** - identify duplicate files в large datasets
- **Timeline analysis** - verify file creation и modification times
- **Court admissibility** - cryptographically-backed evidence integrity

**Blockchain и Cryptocurrencies:**
- **Block linking** - each block contains hash на previous block
- **Proof-of-work** - mining algorithms based на hash function properties
- **Transaction integrity** - Merkle trees provide efficient verification
- **Address generation** - cryptocurrency addresses derived от public key hashes
- **Smart contracts** - hash-based commitment schemes

**Software Integrity:**
- **Code signing** - hash values ensure software nicht tampered

**🔐 MAC Fundamentals и Security Goals (8 мин)**

**Message Authentication Code Purpose:**
MAC provides both **authentication** (proves message origin) и **integrity** (detects modifications) using shared secret key.

**MAC vs Digital Signatures:**
- **Shared secret** - MAC uses symmetric key, signatures use public/private keys
- **Verification scope** - only parties with key can verify MAC, anyone can verify signature
- **Performance** - MAC operations faster than digital signatures
- **Non-repudiation** - signatures provide non-repudiation, MACs do not
- **Key distribution** - MAC requires secure key sharing

**MAC Security Properties:**

**Message Authentication:**
- **Origin verification** - proves message от someone who knows secret key
- **Insider authentication** - distinguishes між legitimate parties и outsiders
- **Key-dependent** - without key, cannot generate valid MAC
- **Symmetric property** - all parties који key can generate и verify

**Message Integrity:**
- **Modification detection** - any change в message или MAC detected
- **Bit-level sensitivity** - single bit change invalidates MAC
- **Complete coverage** - MAC protects entire message content
- **Tamper evidence** - provides cryptographic proof на integrity

**Unforgeability:**
- **Existential unforgeability** - cannot create valid MAC за new message
- **Key recovery resistance** - observing MACs doesn't reveal secret key
- **Selective forgery resistance** - cannot create MAC за chosen message
- **Adaptive security** - secure even когато attacker chooses messages adaptively

**🔧 HMAC Construction и Security (8 мин)**

**HMAC (Hash-based Message Authentication Code) Design:**

**HMAC Algorithm:**
```
HMAC(K, M) = H((K ⊕ opad) || H((K ⊕ ipad) || M))

Where:
K = secret key
M = message  
H = hash function (SHA-256, SHA-3, etc.)
opad = outer padding (0x5c repeated)
ipad = inner padding (0x36 repeated)
|| = concatenation
⊕ = XOR operation
```

**Key Preprocessing:**
- **Key length adjustment** - если key longer than block size, hash it first
- **Padding to block size** - pad key с zeros to match hash block size
- **Standard key lengths** - typically 256 bits (32 bytes) за strong security

**HMAC Security Advantages:**
- **Provable security** - security reduces to underlying hash function
- **Nested construction** - двойна hashing prevents length extension attacks
- **Key separation** - inner и outer keys derived от master key
- **Standardization** - RFC 2104, FIPS 198, widespread implementation

**HMAC Variants:**
- **HMAC-SHA256** - most commonly used, good performance и security
- **HMAC-SHA512** - higher security level, better performance на 64-bit systems
- **HMAC-SHA3** - modern alternative with different security assumptions
- **HMAC-MD5** - legacy, не recommended due to MD5 weaknesses

#### **Час 14: Hash-based Key Derivation (40 мин)**

**Теоретична част (22 мин):**

**🔑 Key Derivation Function (KDF) Fundamentals (10 мин)**

**Purpose на Key Derivation:**
Transform low-entropy secret (password, shared key) в high-entropy cryptographic keys suitable за encryption, authentication, и other cryptographic operations.

**Key Derivation Requirements:**
- **Deterministic output** - same input always produces same derived key
- **High entropy** - output should be cryptographically random
- **Appropriate length** - generate keys на required length за target algorithm
- **One-way function** - computationally infeasible to reverse
- **Salt support** - incorporate random values to prevent rainbow table attacks

**HKDF (HMAC-based Key Derivation Function):**

**HKDF Process (RFC 5869):**
1. **Extract phase** - PRK = HKDF-Extract(salt, IKM)
2. **Expand phase** - OKM = HKDF-Expand(PRK, info, L)

**Extract Phase:**
```
PRK = HMAC-Hash(salt, IKM)

Where:
PRK = Pseudo-Random Key
salt = optional salt value (random bytes)
IKM = Input Key Material (low-entropy secret)
```

**Expand Phase:**
```
OKM = HKDF-Expand(PRK, info, L)

Where:
OKM = Output Key Material (derived keys)
info = optional context information
L = length на desired output в octets
```

**HKDF Security Benefits:**
- **Information-theoretic security** - based на randomness extraction theory
- **Flexibility** - can derive multiple keys от single master secret
- **Context separation** - different contexts produce different keys
- **Standardization** - widely adopted в TLS, Signal, WireGuard

**🔒 Password-Based Key Derivation (8 мин)**

**PBKDF2 (Password-Based Key Derivation Function 2):**

**PBKDF2 Algorithm:**
```
DK = PBKDF2(Password, Salt, IterationCount, DerivedKeyLength)

Process:
1. Initialize: U₁ = HMAC(Password, Salt || 0x00000001)
2. Iterate: U₂ = HMAC(Password, U₁), U₃ = HMAC(Password, U₂), ...
3. XOR results: DK = U₁ ⊕ U₂ ⊕ ... ⊕ U_c
```

**PBKDF2 Parameters:**
- **Iteration count** - typically 100,000+ za protection against brute force
- **Salt length** - minimum 128 bits (16 bytes) za rainbow table protection  
- **Output length** - match requirements на target cryptographic algorithm
- **Hash function** - SHA-256 commonly used, SHA-1 legacy

**Advanced Key Derivation Functions:**

**Argon2 (Winner на Password Hashing Competition):**
- **Memory-hard function** - requires significant memory, prevents ASIC attacks
- **Three variants:** Argon2d, Argon2i, Argon2id (recommended)
- **Configurable parameters:** time, memory, parallelism
- **Modern standard** - preferred за new password hashing applications

**scrypt:**
- **Memory-hard function** - designed to be expensive за specialized hardware
- **Tunable parameters** - CPU cost, memory cost, parallelization
- **Cryptocurrency usage** - used в Litecoin и other cryptocurrencies
- **Good performance** - efficient на general-purpose hardware

**bcrypt:**
- **Adaptive function** - cost parameter increases over time
- **Blowfish-based** - uses modified Blowfish key setup
- **Wide adoption** - supported в many programming frameworks
- **Time-tested** - proven security record в production systems

**Теоретична част (22 мин):**

**🔑 Symmetric Encryption Principles (8 мин)**

**Core Concept:**
Symmetric cryptography използва same key за encryption и decryption. Both sender и receiver must possess identical secret key за secure communication.

**Key Characteristics:**
- **Single shared key** - same key encrypts и decrypts data
- **Fast processing** - computationally efficient за large data volumes
- **Key distribution challenge** - secure key sharing between parties
- **Scalability issues** - n(n-1)/2 keys needed за n participants
- **Perfect за bulk encryption** - files, databases, network traffic

**Historical Context:**
- **Caesar Cipher (50 BC)** - simple character substitution
- **Enigma Machine (WWII)** - mechanical encryption device
- **DES (1977)** - first computer-age encryption standard
- **AES (2001)** - current symmetric encryption standard

**🔐 Block vs Stream Ciphers (7 мин)**

**Block Ciphers:**
- **Fixed-size blocks** - encrypt data в fixed chunks (typically 128 bits)
- **Examples:** AES (128-bit blocks), DES (64-bit blocks), Blowfish
- **Padding required** - handle data не exactly divisible by block size
- **Mode-dependent security** - different modes provide different security properties
- **Use cases:** File encryption, database encryption, secure storage

**Stream Ciphers:**
- **Bit-by-bit encryption** - encrypt data one bit или byte at a time
- **Pseudorandom keystream** - generate continuous stream на encryption keys
- **Examples:** RC4, ChaCha20, Salsa20
- **No padding needed** - handle arbitrary data lengths naturally
- **Use cases:** Real-time communications, wireless networks, streaming media

**🛡️ Block Cipher Modes (7 мин)**

**Electronic Codebook (ECB) Mode:**
- **Direct encryption** - each block encrypted independently
- **Security weakness** - identical plaintext blocks produce identical ciphertext
- **Visual patterns** - images show clear patterns when encrypted със ECB
- **Not recommended** за most applications due to pattern leakage

**Cipher Block Chaining (CBC) Mode:**
- **Block chaining** - each block XORed със previous ciphertext block
- **Initialization Vector (IV)** - random first block за prevent pattern recognition
- **Sequential processing** - cannot parallelize encryption
- **Error propagation** - single bit error affects multiple blocks

**Counter (CTR) Mode:**
- **Stream-like behavior** - turns block cipher в stream cipher
- **Parallelizable** - blocks can be encrypted independently
- **Random access** - can decrypt any block без decrypting previous blocks
- **No error propagation** - single bit errors affect только that bit

**Galois/Counter Mode (GCM):**
- **Authenticated encryption** - provides both confidentiality и integrity
- **High performance** - designed за fast implementation
- **AEAD (Authenticated Encryption със Associated Data)** - protects metadata
- **Modern standard** - used в TLS 1.3, IPSec, wireless protocols

#### **Час 2: Advanced Encryption Standard (AES) (40 мин)**

**Теоретична част (25 мин):**

**🏆 AES History и Selection (8 мин)**

**NIST AES Competition (1997-2000):**
- **Background:** DES becoming vulnerable due to key length (56 bits)
- **Requirements:** 128, 192, 256-bit keys; secure, efficient, flexible
- **15 candidates** submitted from worldwide cryptographic community
- **5 finalists:** Rijndael (winner), Serpent, Twofish, RC6, MARS
- **Selection criteria:** Security, performance, implementation flexibility

**Rijndael Algorithm Victory:**
- **Designed by:** Belgian cryptographers Joan Daemen и Vincent Rijmen
- **Key advantages:** Simple design, fast software/hardware implementation
- **Scalability:** Variable block и key sizes (standardized to 128-bit blocks)
- **Mathematical elegance:** Based на finite field arithmetic
- **Adoption:** Became AES standard в 2001, still unbroken today

**🔢 AES Mathematical Foundation (9 мин)**

**Finite Field Arithmetic (GF(2^8)):**
- **Galois field operations** - addition, multiplication в finite field
- **Polynomial representation** - bytes treated като polynomials
- **Irreducible polynomial** - x^8 + x^4 + x^3 + x + 1
- **S-box construction** - multiplicative inverse в GF(2^8)

**AES Operations:**

**SubBytes Transformation:**
- **S-box substitution** - each byte replaced with S-box value
- **Non-linear operation** - provides confusion property
- **Resistant to differential cryptanalysis** - carefully designed S-box
- **Invertible transformation** - enables decryption

**ShiftRows Transformation:**
- **Byte transposition** - circular left shift на each row
- **Row 0:** No shift, **Row 1:** 1 byte, **Row 2:** 2 bytes, **Row 3:** 3 bytes
- **Diffusion property** - spreads influence на each byte
- **Simple operation** - efficient в software и hardware

**MixColumns Transformation:**
- **Matrix multiplication** в GF(2^8) - provides excellent diffusion
- **Column-wise operation** - each 4-byte column mixed independently
- **Linear transformation** - complements non-linear SubBytes
- **Omitted в final round** - security optimization

**AddRoundKey:**
- **XOR operation** - combines state със round key
- **Key schedule** - derives round keys от main key
- **Simplest operation** - but provides key-dependent transformation

**🔄 AES Key Schedule и Security (8 мин)**

**Key Expansion Process:**
- **AES-128:** 10 rounds, needs 11 round keys (176 bytes total)
- **AES-192:** 12 rounds, needs 13 round keys (208 bytes total)
- **AES-256:** 14 rounds, needs 15 round keys (240 bytes total)

**Security Properties:**
- **Confusion:** S-box makes each bit на ciphertext depend на several plaintext bits
- **Diffusion:** ShiftRows и MixColumns ensure changes propagate throughout state
- **Key-dependent:** Each round key provides unique transformation
- **Proven security:** Resistant to known cryptanalytic attacks

**Implementation Security:**
- **Side-channel attacks:** Timing, power analysis, electromagnetic emissions
- **AES-NI instructions:** Hardware acceleration on modern processors
- **Constant-time implementation:** Prevent timing-based information leakage
- **Memory protection:** Secure key storage и handling

#### **Час 3: Stream Ciphers и ChaCha20 (40 мин)**

**Теоретична част (23 мин):**

**🌊 Stream Cipher Fundamentals (8 мин)**

**Stream Cipher Architecture:**
- **Keystream generation** - pseudorandom sequence derived от key
- **XOR operation** - plaintext ⊕ keystream = ciphertext
- **Synchronization** - sender и receiver must maintain same keystream position
- **One-time pad similarity** - approximates perfect security със random keystream

**Advantages over Block Ciphers:**
- **Real-time processing** - no need to buffer complete blocks
- **Variable length** - handle arbitrary message sizes naturally
- **Low latency** - minimal delay в encryption/decryption
- **Hardware efficiency** - simple operations suitable за embedded systems

**Security Requirements:**
- **Unpredictable keystream** - must appear random to attackers
- **Long period** - keystream should не repeat за practical purposes
- **Key sensitivity** - small key changes should completely alter keystream
- **Non-reversibility** - impossible to recover key от keystream

**🚫 RC4 Vulnerabilities и Lessons Learned (8 мин)**

**RC4 Historical Significance:**
- **Designed 1987** by Ron Rivest за RSA Security
- **Trade secret initially** - reverse engineered и published в 1994
- **Widespread adoption** - used в WEP, early WPA, SSL/TLS
- **Simple algorithm** - 256-byte state array, two counters

**Critical Vulnerabilities Discovered:**

**Weak Key Scheduling:**
- **Related key attacks** - similar keys produce related keystreams
- **Key recovery** - possible to derive key от keystream analysis
- **Statistical bias** - keystream не truly random
- **First bytes weakness** - initial keystream bytes show patterns

**Real-World Exploits:**
- **WEP cracking** - RC4 weaknesses enable rapid Wi-Fi password recovery
- **BEAST attack** - SSL/TLS vulnerabilities using RC4 biases
- **Bar mitzvah attack** - exploits invariance weakness в RC4
- **RFC 7465 (2015)** - officially prohibits RC4 usage в TLS

**🎯 ChaCha20 Modern Stream Cipher (7 мин)**

**ChaCha20 Design Principles:**
- **Daniel J. Bernstein design** - created by respected cryptographer
- **ARX operations** - Addition, Rotation, XOR (simple, secure operations)
- **Quarter-round function** - core mixing operation
- **512-bit state** - 256-bit key, 64-bit nonce, 64-bit counter

**Technical Advantages:**
- **Proven security** - extensive cryptanalytic analysis
- **Performance** - faster than AES на software-only platforms
- **Side-channel resistance** - constant-time implementation natural
- **Parallelizable** - can generate keystream blocks independently

**ChaCha20-Poly1305 AEAD:**
- **Authenticated encryption** - combines ChaCha20 cipher с Poly1305 MAC
- **TLS 1.3 adoption** - preferred cipher suite в modern TLS
- **Google adoption** - used throughout Google services
- **Internet standard** - RFC 8439 defines ChaCha20-Poly1305

**Real-World Applications:**
- **Signal messenger** - end-to-end encrypted messaging
- **WireGuard VPN** - modern VPN protocol
- **Android encryption** - full-disk encryption на mobile devices
- **Cryptocurrency** - various blockchain implementations

**Теоретична част (22 мин):**

**🔧 RSA Implementation Details (10 мин)**

**Key Generation Process:**
1. **Prime selection:** Choose large primes p и q (1024+ bits each)
2. **Modulus calculation:** n = p × q (public modulus)
3. **Totient calculation:** φ(n) = (p-1)(q-1)
4. **Public exponent:** Choose e coprime to φ(n), commonly e = 65537
5. **Private exponent:** Calculate d such that e × d ≡ 1 (mod φ(n))
6. **Key validation:** Verify mathematical relationships

**Prime Generation Challenges:**
- **Primality testing:** Miller-Rabin test за large numbers
- **Strong primes:** Additional mathematical properties за enhanced security
- **Side-channel resistance:** Prevent timing attacks during generation
- **Entropy requirements:** Sufficient randomness за unpredictable primes

**Padding Schemes и Security:**

**PKCS#1 v1.5 Padding:**
- **Structure:** 0x00 || 0x02 || PS || 0x00 || M
- **Purpose:** Prevent mathematical attacks на raw RSA
- **Vulnerability:** Padding oracle attacks possible
- **Legacy status:** Still widely used но not recommended

**OAEP (Optimal Asymmetric Encryption Padding):**
- **Enhanced security:** Provably secure under random oracle model  
- **Structure:** More complex padding с random elements
- **CCA security:** Resistant to chosen-ciphertext attacks
- **Current standard:** Recommended за new RSA implementations

**🚨 RSA Vulnerabilities и Mitigations (12 мин)**

**Mathematical Attack Vectors:**

**Small Exponent Attacks:**
- **Low public exponent:** e = 3 enables certain mathematical attacks
- **Broadcast attack:** Same message encrypted за multiple recipients
- **Mitigation:** Use standard exponent e = 65537, proper padding

**Timing Attacks:**
- **Implementation variation:** Decryption time varies based на private key bits
- **Side-channel leakage:** Statistical analysis reveals private key
- **Mitigation:** Constant-time implementation, blinding techniques

**Fault Injection Attacks:**
- **Hardware manipulation:** Induce computational errors during operations
- **Key recovery:** Analyze faulty outputs to derive private key
- **Mitigation:** Hardware security modules, error detection

**Common Implementation Mistakes:**

**Weak Random Number Generation:**
- **Debian OpenSSL bug (2008):** Predictable random numbers
- **Shared prime factors:** Different key pairs sharing prime factors
- **Impact:** Private key recovery через greatest common divisor (GCD)
- **Prevention:** Quality entropy sources, prime uniqueness verification

**Poor Key Management:**
- **Key reuse:** Same key pair used за encryption и signatures
- **Long-lived keys:** Keys never rotated или updated
- **Insecure storage:** Private keys stored без protection
- **Solution:** Proper key lifecycle management, hardware security modules

**Elliptic Curve Definition:**
Mathematical equation: y² = x³ + ax + b (mod p)

**Key Mathematical Properties:**
- **Point addition:** Geometric operation на curve points
- **Scalar multiplication:** k × P = P + P + ... + P (k times)
- **Discrete logarithm problem:** Given P и Q = k × P, find k
- **Computational complexity:** Much harder than integer factorization

**Point Addition Rules:**
1. **Identity element:** Point at infinity (O) acts като additive identity
2. **Point negation:** P + (-P) = O
3. **Distinct points:** Line through P₁ и P₂ intersects curve at third point
4. **Point doubling:** Tangent line construction за P + P

**Curve Parameters:**
- **Prime field curves:** Operations modulo large prime p
- **Binary field curves:** Operations в GF(2ᵐ) finite fields
- **Standardized curves:** NIST P-256, P-384, P-521
- **Alternative curves:** Curve25519, secp256k1 (Bitcoin)

**🔒 ECC Security Advantages (8 мин)**

- **Smaller key sizes:** 256-bit ECC ≈ 3072-bit RSA security
- **Faster operations:** Point multiplication more efficient than exponentiation
- **Lower bandwidth:** Smaller certificates, signatures, keys
- **Battery friendly:** Important за mobile и IoT devices

**Security Strength:**
- **Exponential security:** Each additional bit doubles security
- **No subexponential attacks:** Unlike factorization-based systems
- **Clean mathematical structure:** Well-understood mathematical foundation
- **Side-channel resistance:** More naturally resistant to timing attacks

**Standardization Status:**
- **NIST approval:** Recommended by major standards bodies
- **Industry adoption:** TLS, IPSec, Bitcoin, many cryptographic protocols

**🔐 RSA Signatures vs Encryption (7 мин)**

**RSA Signature Generation:**
- **Hash message:** M → H(M) using SHA-256
- **Sign hash:** S = H(M)^d mod n (private key operation)
- **Signature:** (M, S) transmitted together
- **Security:** Based на private key secrecy

**RSA Signature Verification:**  
- **Recover hash:** H'(M) = S^e mod n (public key operation)
- **Compute hash:** H(M) directly от received message
- **Compare:** If H'(M) = H(M), signature valid
- **Result:** Authentication и integrity confirmed

**Key Differences от Encryption:**
- **Direction reversal:** Signatures use private key first, public key second
- **Hash requirement:** Sign hash, не entire message (efficiency)
- **Public verification:** Anyone can verify signature validity
- **Legal implications:** Signatures have stronger legal standing

**ECDSA (Elliptic Curve Digital Signature Algorithm):**
- **Efficiency:** Smaller signatures, faster verification
- **Randomness critical:** Requires unique random number за each signature
- **Nonce reuse vulnerability:** Same random number reveals private key
