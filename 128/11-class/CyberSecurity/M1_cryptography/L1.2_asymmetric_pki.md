#### **Час 6: Public Key Cryptography Fundamentals (40 мин)**

**Теоретична част (25 мин):**

**🗝️ Asymmetric Cryptography Revolution (10 мин)**

**The Key Distribution Problem Solution:**
Traditional symmetric cryptography required secure channel за key sharing. Public key cryptography eliminates this requirement through mathematical innovation.

**Revolutionary Concept (1976):**
- **Whitfield Diffie и Martin Hellman** introduced concept на public-key cryptography
- **Two related keys:** Public key (widely distributed), Private key (kept secret)
- **Mathematical relationship:** Operations with одна key can be reversed only със the other
- **Security foundation:** Based на computationally difficult mathematical problems

**Core Principles:**
- **Public key encryption:** Anyone can encrypt messages для specific recipient
- **Digital signatures:** Prove message authenticity и non-repudiation
- **Key establishment:** Securely agree на shared symmetric keys
- **No prior shared secrets:** Communicate securely без previous contact

**🔢 Mathematical Foundations (8 мин)**

**One-Way Functions:**
- **Easy computation:** Forward direction computationally simple
- **Difficult inversion:** Reverse direction computationally infeasible
- **Trapdoor property:** Secret information enables easy inversion
- **Security basis:** Asymmetric algorithms rely на these properties

**RSA Mathematical Basis:**
- **Integer factorization problem:** Easy to multiply primes, hard to factor products
- **Large prime numbers:** 1024, 2048, 3072-bit numbers commonly used
- **Modular arithmetic:** Operations performed modulo large composite number
- **Euler's theorem:** Mathematical foundation за RSA correctness

**Elliptic Curve Mathematics:**
- **Elliptic curve discrete logarithm problem (ECDLP):** Harder than integer factorization
- **Smaller key sizes:** 256-bit ECC equivalent to 3072-bit RSA security
- **Efficient operations:** Faster computation с lower power consumption
- **Mobile-friendly:** Ideal за resource-constrained devices

**Quantum Computing Threats:**
- **Shor's algorithm:** Efficiently factors integers и solves discrete logarithms
- **Timeline concerns:** Large quantum computers may break current algorithms
- **Post-quantum cryptography:** Research в quantum-resistant algorithms
- **Migration planning:** Organizations preparing за quantum-safe transition

**🔐 Public Key Operations (7 мин)**

**Encryption Process:**
1. **Key pair generation:** Recipient creates public/private key pair
2. **Public key distribution:** Public key shared openly
3. **Message encryption:** Sender encrypts с recipient's public key
4. **Secure transmission:** Ciphertext sent over insecure channel
5. **Decryption:** Only recipient can decrypt с private key

**Digital Signature Process:**
1. **Message hashing:** Original message reduced to fixed-size digest
2. **Signature generation:** Private key signs hash (not entire message)
3. **Signature transmission:** Message и signature sent together
4. **Signature verification:** Public key verifies signature authenticity
5. **Integrity confirmation:** Modified messages fail signature verification

**Hybrid Cryptography:**
- **Performance optimization:** Public key cryptography slower than symmetric
- **Best practices:** Use public key за symmetric key exchange
- **Session keys:** Temporary symmetric keys за bulk data encryption
- **Perfect forward secrecy:** New session keys за each communication session

#### **Час 7: RSA Implementation и Security (40 мин)**

**Теоретична част (22 мин):**

**🔧 RSA Implementation Details (10 мин)**

**Key Generation Process:**
1. **Prime selection:** Choose large primes p и q (1024+ bits each)
2. **Modulus calculation:** n = p × q (public modulus)
3. **Totient calculation:** φ(n) = (p-1)(q-1)
4. **Public exponent:** Choose e coprime to φ(n), commonly e = 65537
5. **Private exponent:** Calculate d such that e × d ≡ 1 (mod φ(n))
6. **Key validation:** Verify mathematical relationships

**Prime Generation Challenges:**
- **Primality testing:** Miller-Rabin test за large numbers
- **Strong primes:** Additional mathematical properties за enhanced security
- **Side-channel resistance:** Prevent timing attacks during generation
- **Entropy requirements:** Sufficient randomness за unpredictable primes

**Padding Schemes и Security:**

**PKCS#1 v1.5 Padding:**
- **Structure:** 0x00 || 0x02 || PS || 0x00 || M
- **Purpose:** Prevent mathematical attacks на raw RSA
- **Vulnerability:** Padding oracle attacks possible
- **Legacy status:** Still widely used но not recommended

**OAEP (Optimal Asymmetric Encryption Padding):**
- **Enhanced security:** Provably secure under random oracle model
- **Structure:** More complex padding с random elements
- **CCA security:** Resistant to chosen-ciphertext attacks
- **Current standard:** Recommended за new RSA implementations

**🚨 RSA Vulnerabilities и Mitigations (12 мин)**

**Mathematical Attack Vectors:**

**Small Exponent Attacks:**
- **Low public exponent:** e = 3 enables certain mathematical attacks
- **Broadcast attack:** Same message encrypted за multiple recipients
- **Mitigation:** Use standard exponent e = 65537, proper padding

**Timing Attacks:**
- **Implementation variation:** Decryption time varies based на private key bits
- **Side-channel leakage:** Statistical analysis reveals private key
- **Mitigation:** Constant-time implementation, blinding techniques

**Fault Injection Attacks:**
- **Hardware manipulation:** Induce computational errors during operations
- **Key recovery:** Analyze faulty outputs to derive private key
- **Mitigation:** Hardware security modules, error detection

**Common Implementation Mistakes:**

**Weak Random Number Generation:**
- **Debian OpenSSL bug (2008):** Predictable random numbers
- **Shared prime factors:** Different key pairs sharing prime factors
- **Impact:** Private key recovery через greatest common divisor (GCD)
- **Prevention:** Quality entropy sources, prime uniqueness verification

**Poor Key Management:**
- **Key reuse:** Same key pair used за encryption и signatures
- **Long-lived keys:** Keys never rotated или updated
- **Insecure storage:** Private keys stored без protection
- **Solution:** Proper key lifecycle management, hardware security modules

#### **Час 8: Elliptic Curve Cryptography (ECC) (40 мин)**

**Теоретична част (24 мин):**

**📐 Elliptic Curve Mathematics (10 мин)**

**Elliptic Curve Definition:**
Mathematical equation: y² = x³ + ax + b (mod p)

**Key Mathematical Properties:**
- **Point addition:** Geometric operation на curve points
- **Scalar multiplication:** k × P = P + P + ... + P (k times)
- **Discrete logarithm problem:** Given P и Q = k × P, find k
- **Computational complexity:** Much harder than integer factorization

**Point Addition Rules:**
1. **Identity element:** Point at infinity (O) acts като additive identity
2. **Point negation:** P + (-P) = O
3. **Distinct points:** Line through P₁ и P₂ intersects curve at third point
4. **Point doubling:** Tangent line construction за P + P

**Curve Parameters:**
- **Prime field curves:** Operations modulo large prime p
- **Binary field curves:** Operations в GF(2ᵐ) finite fields
- **Standardized curves:** NIST P-256, P-384, P-521
- **Alternative curves:** Curve25519, secp256k1 (Bitcoin)

**🔒 ECC Security Advantages (8 мин)**

**Computational Efficiency:**
- **Smaller key sizes:** 256-bit ECC ≈ 3072-bit RSA security
- **Faster operations:** Point multiplication more efficient than exponentiation
- **Lower bandwidth:** Smaller certificates, signatures, keys
- **Battery friendly:** Important за mobile и IoT devices

**Security Strength:**
- **Exponential security:** Each additional bit doubles security
- **No subexponential attacks:** Unlike factorization-based systems
- **Clean mathematical structure:** Well-understood mathematical foundation
- **Side-channel resistance:** More naturally resistant to timing attacks

**Standardization Status:**
- **NIST approval:** Recommended by major standards bodies
- **Industry adoption:** TLS, IPSec, Bitcoin, many cryptographic protocols
- **Government use:** NSA Suite B cryptography (though later revised)
- **International standards:** ISO, ANSI, IEEE standardization

**⚡ ECC Implementation Considerations (6 мин)**

**Curve Selection Criteria:**
- **Security level:** Match curve strength to application requirements
- **Performance characteristics:** Some curves optimized за specific operations
- **Patent issues:** Some curves have intellectual property restrictions
- **Standardization status:** Prefer well-analyzed, standardized curves

**Common Curves:**

**NIST P-256 (secp256r1):**
- **Security:** 128-bit equivalent strength
- **Usage:** TLS, government applications, general purpose
- **Advantages:** Widely supported, extensively analyzed
- **Concerns:** NSA involvement в standardization process

**Curve25519:**
- **Design:** Daniel J. Bernstein's high-performance curve
- **Security:** Designed to avoid implementation pitfalls
- **Performance:** Faster than NIST curves на many platforms
- **Usage:** Signal, WireGuard, modern applications

**secp256k1:**
- **Bitcoin usage:** Cryptocurrency и blockchain applications
- **Characteristics:** Koblitz curve с specific mathematical properties
- **Performance:** Efficient signature verification
- **Community:** Strong development community, extensive testing

#### **Час 9: Digital Signatures и Non-Repudiation (40 мин)**

**Теоретична част (23 мин):**

**✍️ Digital Signature Fundamentals (8 мин)**

**Digital Signature Properties:**
- **Authentication:** Proves message origin (who sent it)
- **Integrity:** Detects any message modifications
- **Non-repudiation:** Sender cannot deny sending message
- **Timestamping:** Can include time на signature creation

**Digital vs Handwritten Signatures:**
- **Uniqueness:** Each signature mathematically unique to message
- **Verification:** Anyone can verify без revealing private key
- **Forgery resistance:** Computationally infeasible to forge
- **Legal validity:** Recognized в many jurisdictions worldwide

**Signature Process:**
1. **Message hashing:** Create fixed-size digest на message
2. **Private key signing:** Apply private key to hash
3. **Signature attachment:** Combine signature със message
4. **Public verification:** Anyone can verify usando public key
5. **Result interpretation:** Valid/invalid signature determination

**🔐 RSA Signatures vs Encryption (7 мин)**

**RSA Signature Generation:**
- **Hash message:** M → H(M) using SHA-256
- **Sign hash:** S = H(M)^d mod n (private key operation)
- **Signature:** (M, S) transmitted together
- **Security:** Based на private key secrecy

**RSA Signature Verification:**
- **Recover hash:** H'(M) = S^e mod n (public key operation)
- **Compute hash:** H(M) directly от received message
- **Compare:** If H'(M) = H(M), signature valid
- **Result:** Authentication и integrity confirmed

**Key Differences от Encryption:**
- **Direction reversal:** Signatures use private key first, public key second
- **Hash requirement:** Sign hash, не entire message (efficiency)
- **Public verification:** Anyone can verify signature validity
- **Legal implications:** Signatures have stronger legal standing

**ECDSA (Elliptic Curve Digital Signature Algorithm):**
- **Efficiency:** Smaller signatures, faster verification
- **Randomness critical:** Requires unique random number за each signature
- **Nonce reuse vulnerability:** Same random number reveals private key
- **Mathematical complexity:** Based на elliptic curve discrete logarithm

#### **Час 10: Public Key Infrastructure (PKI) Fundamentals (40 мин)**

**Теоретична част (25 мин):**

**🏗️ PKI Architecture Components (10 мин)**

**Certificate Authority (CA) Hierarchy:**

**Root Certificate Authority:**
- **Highest trust level** - self-signed certificate, ultimate trust anchor
- **Offline storage** - kept в highly secure, air-gapped environment
- **Long validity** - typically 10-30 years
- **Minimal usage** - only signs intermediate CA certificates
- **Critical protection** - compromise affects entire PKI

**Intermediate Certificate Authorities:**
- **Operational signing** - issues end-entity certificates daily
- **Online presence** - accessible for certificate validation
- **Shorter validity** - typically 1-5 years
- **Delegated trust** - derives authority от root CA
- **Revocable** - can be revoked if compromised

**End-Entity Certificates:**
- **User certificates** - individuals, devices, servers
- **Short validity** - typically 1-2 years
- **Specific purposes** - encryption, signing, authentication
- **Revocable** - can be revoked anytime if needed
- **Leaf nodes** - cannot sign other certificates

**Registration Authority (RA):**
- **Identity verification** - validates certificate applicant identity
- **Policy enforcement** - ensures compliance с certificate policies
- **Request processing** - handles certificate requests и renewals
- **Local presence** - може да be geographically distributed
- **CA separation** - allows CA to focus на cryptographic operations

**🔖 Digital Certificates Structure (8 мин)**

**X.509 Certificate Format (Version 3):**

**Basic Certificate Fields:**
```
Version: v3 (most common version)
Serial Number: Unique identifier за this certificate
Signature Algorithm: RSA-SHA256, ECDSA-SHA256, etc.
Issuer: Certificate Authority που issued certificate
Validity Period: Not Before и Not After dates
Subject: Entity this certificate identifies
Subject Public Key: Public key и algorithm parameters
```

**X.509v3 Extensions (Critical za modern PKI):**

**Key Usage Extension:**
- **Digital Signature** - can create digital signatures
- **Key Encipherment** - can encrypt symmetric keys
- **Data Encipherment** - can encrypt user data directly
- **Key Agreement** - can perform key exchange operations
- **Certificate Signing** - can sign other certificates (CA only)
- **CRL Signing** - can sign certificate revocation lists

**Extended Key Usage:**
- **Server Authentication** - TLS/SSL server certificates
- **Client Authentication** - TLS/SSL client certificates
- **Code Signing** - software code signing certificates
- **Email Protection** - S/MIME email certificates
- **Time Stamping** - trusted timestamping certificates

**Subject Alternative Names (SAN):**
- **DNS Names** - www.example.com, mail.example.com

- **Email Addresses** - user@example.com
- **URIs** - https://example.com/service

**🔄 Certificate Lifecycle Management (7 мин)**

**Certificate Request Process:**
1. **Key pair generation** - applicant creates public/private key pair

3. **Identity verification** - RA validates applicant identity
4. **Certificate issuance** - CA signs и issues certificate
5. **Certificate distribution** - certificate provided to applicant

**Certificate Validation:**
- **Signature verification** - verify CA's digital signature
- **Validity period** - check current date против certificate dates
- **Revocation checking** - verify certificate not revoked
- **Chain validation** - validate entire certificate chain to root
- **Key usage validation** - verify certificate appropriate за intended use

**Certificate Revocation:**

**Certificate Revocation Lists (CRLs):**
- **Periodic publication** - CAs publish lists на revoked certificates
- **Serial number listing** - revoked certificates identified by serial number
- **Revocation reasons** - key compromise, superseded, cessation на operation
- **Distribution points** - specified locations за downloading CRLs

**Online Certificate Status Protocol (OCSP):**
- **Real-time checking** - query certificate status онлайн
- **Individual queries** - check specific certificate status
- **Performance benefits** - avoid downloading large CRLs
- **Privacy concerns** - CA learns which certificates being validated
- **OCSP Stapling** - server provides OCSP response directly

**📋 Certificate Transparency и Monitoring (6 мин)**

**Certificate Transparency (CT) Framework:**

**Purpose и Goals:**
- **Detect mis-issued certificates** - identify unauthorized certificates
- **Increase accountability** - make CA operations more transparent
- **Enable monitoring** - allow domain owners to monitor their domains
- **Improve security** - detect attacks using fraudulent certificates

**CT Log Architecture:**
- **Append-only logs** - certificates added но never removed
- **Merkle tree structure** - cryptographically verifiable log integrity
- **Public accessibility** - anyone can query и monitor logs
- **Distributed operation** - multiple independent log operators
- **Browser enforcement** - browsers require CT за new certificates

**CT Process Flow:**
1. **Certificate submission** - CA submits certificate to CT log
2. **Signed Certificate Timestamp (SCT)** - log returns proof на submission
3. **Certificate issuance** - CA issues certificate със embedded SCT
4. **Browser validation** - browser verifies SCT during TLS handshake
5. **Monitoring** - domain owners monitor logs за unauthorized certificates

**Certificate Monitoring Tools:**

**Certificate Transparency Monitoring:**
- **crt.sh** - public CT log search engine
- **Facebook Certificate Transparency Monitoring** - automated alerting
- **Censys** - comprehensive internet-wide certificate analysis
- **SSLMate Certificate Transparency Search** - domain-focused monitoring
