#### **Час 2: Data Loss Prevention (DLP) Technologies (40 мин)**

**Теоретична част (24 мин):**

**🛡️ DLP Architecture и Components (10 мин)**

**Data Loss Prevention Purpose:**
DLP solutions monitor, detect, и prevent unauthorized transmission, access, или use на sensitive data across the organization.

**DLP Deployment Models:**

**Network-Based DLP:**
- **Gateway deployment** - monitors all network traffic leaving organization
- **Protocol support** - email, web, FTP, instant messaging
- **Content inspection** - deep packet inspection за sensitive data patterns
- **Policy enforcement** - block, quarantine, или encrypt suspicious transmissions
- **Incident logging** - comprehensive audit trail за all policy violations

**Endpoint DLP:**
- **Agent-based** - software installed на user workstations и servers
- **File monitoring** - track file operations, copying, deletion
- **Application control** - monitor data usage в specific applications
- **Removable media** - control USB drives, CD/DVD burning
- **Screen capture prevention** - block screenshots на sensitive information

**Storage DLP:**
- **Data discovery** - scan file systems и databases за sensitive content
- **Content classification** - automatically categorize discovered data
- **Access monitoring** - track who accesses sensitive files
- **Encryption enforcement** - automatically encrypt sensitive data at rest
- **Compliance reporting** - generate reports за regulatory requirements

**Cloud DLP:**
- **SaaS integration** - monitor cloud applications (Office 365, Google Workspace)
- **API connectivity** - integrate със cloud storage services
- **Shadow IT discovery** - identify unauthorized cloud service usage
- **Data governance** - enforce policies across hybrid environments

**🔍 DLP Detection Technologies (8 мин)**

**Content Analysis Techniques:**

**Pattern Matching:**
- **Regular expressions** - detect SSNs, credit cards, national IDs
- **Keyword dictionaries** - search за specific terms или phrases
- **Data formats** - identify structured data patterns
- **Custom patterns** - organization-specific sensitive data formats

*Bulgarian Examples:*
```

```

**Machine Learning Classification:**
- **Document classification** - categorize documents by content type
- **Language detection** - identify document language
- **Sentiment analysis** - detect confidential communications
- **Anomaly detection** - identify unusual data access patterns
- **User behavior analytics** - detect insider threat indicators

**Fingerprinting Techniques:**
- **Exact data matching** - match against known sensitive databases
- **Partial document matching** - identify excerpts от confidential documents
- **Statistical fingerprinting** - analyze document statistical properties
- **Indexed document matching** - search against document repositories

**Contextual Analysis:**
- **Metadata examination** - file properties, creation dates, authors
- **Communication context** - email recipients, message content
- **User role analysis** - compare access против job responsibilities
- **Time-based patterns** - detect unusual access times або frequency

**📊 DLP Policy Framework (6 мин)**

**Policy Components:**

**Data Classification Integration:**
```
Policy Structure:
IF (Data Classification = "HIGHLY CONFIDENTIAL")
AND (Action = "Email External")
THEN (Block + Alert Security Team)

IF (Data Classification = "CONFIDENTIAL") 
AND (Action = "USB Transfer")
AND (User Role ≠ "Authorized Data Handler")
THEN (Block + Notify Manager)
```

**Policy Actions:**
- **Block** - prevent the action completely
- **Quarantine** - isolate the data for review
- **Encrypt** - automatically encrypt the transmission
- **Notify** - alert users или administrators
- **Log** - record the incident за audit purposes
- **Redirect** - route to secure communication channel

**Policy Enforcement Levels:**
```
Monitor Mode: Log activities, no blocking
- Purpose: Baseline establishment, policy tuning
- Duration: Initial deployment phase
- Risk: High (no active protection)

Educate Mode: Warn users, allow override
- Purpose: User awareness, gradual enforcement  
- Duration: Training period
- Risk: Medium (user discretion)

Enforce Mode: Block violations, limited overrides
- Purpose: Active protection, business justification required
- Duration: Normal operations
- Risk: Low (strict enforcement)

Lockdown Mode: Block all sensitive data movement
- Purpose: Emergency response, incident containment
- Duration: Crisis situations only
- Risk: Very Low (maximum protection)
```

**User Experience Considerations:**
- **False positive minimization** - tune policies за reduce legitimate blocking
- **Override mechanisms** - business justification procedures  
- **User education** - clear guidance на acceptable behaviors
- **Performance impact** - minimize system slowdown
- **Transparent operations** - users understand protection rationale

#### **Час 3: Data Retention и Disposal Strategies (40 мин)**

**Теоретична част (23 мин):**

**📅 Data Lifecycle Management (9 мин)**

**Information Lifecycle Phases:**

**Active Phase (Hot Data):**
- **Frequent access** - daily operational use
- **High performance** - SSD storage, fast networks
- **Immediate availability** - zero recovery time
- **Current backups** - hourly или daily increments
- **Full functionality** - all features available

**Reference Phase (Warm Data):**
- **Occasional access** - weekly или monthly reference
- **Standard performance** - traditional disk storage
- **Quick availability** - minutes to hours recovery
- **Regular backups** - daily или weekly cycles
- **Limited functionality** - read-only или restricted editing

**Archive Phase (Cold Data):**
- **Rare access** - annual или compliance-driven
- **Lower performance** - tape, optical, cloud archive
- **Longer recovery** - hours to days retrieval time
- **Archive backups** - monthly или quarterly validation
- **Compliance focus** - legal hold, regulatory requirements

**Disposal Phase (End-of-Life):**
- **No business value** - retention period expired
- **Secure destruction** - cryptographic erasure, physical destruction
- **Legal clearance** - no litigation hold или legal requirements
- **Audit documentation** - proof на proper disposal
- **Certificate issuance** - formal destruction verification

**🗑️ Secure Data Disposal Methods (6 мин)**

**Disposal Techniques by Storage Type:**

**Magnetic Storage (HDDs, Tapes):**
- **Overwriting** - multiple passes със random data patterns
- **DoD 5220.22-M standard** - 3-pass overwrite procedure
- **Cryptographic erasure** - destroy encryption keys, render data unreadable
- **Degaussing** - magnetic field destroys data structure
- **Physical destruction** - shredding, crushing, incineration

**Solid State Storage (SSDs, Flash):**
- **ATA Secure Erase** - manufacturer's secure erase command
- **Cryptographic erasure** - preferred method за encrypted drives
- **Physical destruction** - shredding required за highest sensitivity
- **Wear leveling challenges** - data може да remain в unused blocks

**Cloud Data Disposal:**
- **Provider deletion** - rely на cloud provider's deletion procedures
- **Cryptographic erasure** - delete customer-managed encryption keys
- **Data replication** - ensure deletion от all backup locations
- **Contractual guarantees** - require deletion confirmation от provider

**Paper Documents:**
- **Cross-cut shredding** - minimum 4mm x 38mm pieces
- **Industrial shredding** - professional document destruction services
- **Certificate of destruction** - formal proof на secure disposal
- **Chain of custody** - documented handling до destruction

**Disposal Verification:**
```
Disposal Certificate Requirements:
- Asset identification numbers
- Disposal method used
- Date и time на disposal
- Personnel responsible
- Witness signatures
- Destruction facility details
- Environmental disposal method
- Compliance certifications
```

**🗝️ Key Management Strategies (8 мин)**

**Enterprise Key Management Architecture:**

**Hierarchical Key Structure:**
```
Data Encryption Key (DEK):
- Purpose: Encrypt actual data
- Characteristics: AES-256, unique per data set
- Storage: Encrypted by Key Encryption Key
- Rotation: Regular, automated process

Key Encryption Key (KEK):  
- Purpose: Encrypt Data Encryption Keys
- Characteristics: AES-256, hardware protection
- Storage: HSM or secure key vault
- Rotation: Less frequent, manual process

Master Key (MK):
- Purpose: Encrypt Key Encryption Keys
- Characteristics: Hardware-generated, HSM-stored
- Storage: Hardware Security Module only
- Rotation: Rare, planned maintenance events
```
