## **ЧАС 1: SSDLC ПРИНЦИПИ И SECURITY-BY-DESIGN МЕТОДОЛОГИЯ (40 мин)**

### **Преглед на Secure SDLC и рамка (12 мин)**

**Традиционен SDLC срещу Secure SDLC:**

```
Traditional SDLC Approach:
Requirements → Design → Development → Testing → Deployment
                                    ↑
                              Security Added Here
                              (Too Late, Expensive)

Secure SDLC Approach:
Security Requirements → Secure Design → Secure Development → Security Testing → Secure Deployment
    ↓                      ↓              ↓                    ↓                ↓
Security Planning → Threat Modeling → Secure Coding → Penetration Testing → Security Monitoring
```

**Основни принципи на Security-by-Design:**

** 1. Проактивна срещу реактивна сигурност:**
- **Shift-left approach** - интегриране на сигурността от най-ранните фази
- **Цена на поправката** - 100 пъти по-скъпо в production спрямо design фазата
- **Security debt** - натрупване на security проблеми с времето
- **Фокус върху превенцията** - спиране на уязвимостите преди да бъдат създадени

*Реален пример: Имплементацията на SDL от Microsoft намали security уязвимостите с 50%*

** 2. Стратегия за Defense in Depth:**
- **Множество слоеве сигурност** - без единична точка на отказ
- **Резервни защити** - резервни механизми за сигурност
- **Многослойна валидация** - input sanitization + WAF + разрешения на базата данни
- **Постепенна деградация** - системата остава функционална при отказ на слоеве

** 3. Принцип на минималните привилегии:**
- **Минимални права за достъп** - потребителите/процесите получават само необходимите разрешения
- **Just-in-time access** - временно повишаване на привилегиите при нужда
- **Редовни одити на разрешенията** - тримесечни прегледи на достъпа
- **Zero-trust архитектура** - верифицирай всичко, не се доверявай на нищо

** 4. Fail-Safe Defaults:**
- **Сигурно по подразбиране** - приложенията стартират в сигурно състояние
- **Явни разрешения** - всичко е забранено, освен ако не е изрично разрешено
- **Error handling** - грешките не трябва да излагат чувствителна информация
- **Сигурност на конфигурацията** - сигурни настройки по подразбиране в всички системни настройки

## **ЧАС 2: THREAT MODELING И ТЕХНИКИ ЗА ОЦЕНКА НА РИСКА (40 мин)**

### **Основи на Threat Modeling (15 мин)**

**Дефиниция и цел на Threat Modeling:**

Threat modeling е систематичен подход за идентифициране, каталогизиране и приоритизиране на потенциални security заплахи към приложение или система. Целта е да се разберат attack vectors преди deployment на приложението.

**Основни въпроси при Threat Modeling:**
1. **Какво изграждаме?** - Идентификация на активите и разбиране на системата
2. **Какво може да се обърка?** - Идентификация на заплахите и разработване на attack сценарии
3. **Какво трябва да направим по въпроса?** - Разработване на стратегии за намаляване на риска
4. **Свършихме ли добра работа?** - Валидиране и непрекъснато подобряване

**STRIDE категоризация на заплахите:**

```
S - Spoofing Identity:
├── Impersonating legitimate users
├── Using stolen credentials
├── Session hijacking attacks
└── Mitigation: Strong authentication, MFA

T - Tampering with Data:
├── Unauthorized data modification
├── Man-in-the-middle attacks
├── Database injection attacks
└── Mitigation: Data integrity checks, encryption

R - Repudiation:
├── Denying performed actions
├── Log tampering attempts
├── Non-repudiation bypass
└── Mitigation: Audit logging, digital signatures

I - Information Disclosure:
├── Unauthorized data access
├── Data leakage scenarios
├── Privacy violations
└── Mitigation: Access controls, encryption

D - Denial of Service:
├── Resource exhaustion attacks
├── Service availability disruption
├── Performance degradation
└── Mitigation: Rate limiting, load balancing

E - Elevation of Privilege:
├── Unauthorized permission escalation
├── Administrative access bypass
├── Privilege abuse scenarios
└── Mitigation: Least privilege, role separation
```

**DREAD система за оценка на риска:**

**Формула за изчисляване на риска:**
```
Risk Score = (Damage + Reproducibility + Exploitability + Affected Users + Discoverability) / 5

Rating Scale (1-10):
├── 1-3: Low Risk
├── 4-6: Medium Risk
├── 7-8: High Risk
└── 9-10: Critical Risk
```

**Пример за DREAD оценка:**
```
SQL Injection Vulnerability:
├── Damage: 9 (Complete database access)
├── Reproducibility: 8 (Easy to reproduce)
├── Exploitability: 7 (Moderate skill required)
├── Affected Users: 9 (All users affected)
├── Discoverability: 6 (Moderate discovery difficulty)
└── Risk Score: 7.8 (High Risk)
```

### **Оценка на риска и приоритизиране (13 мин)**

**Количествена оценка на риска:**

**Методи за оценяване на активите:**
```
Asset Value Calculation:
├── Replacement Cost: Разходи за пресъздаване на актива
├── Revenue Impact: Пропуснати приходи от компрометиране на актива
├── Regulatory Fines: Потенциални санкции за пробив
├── Reputation Damage: Влияние върху стойността на марката
└── Total Asset Value = Sum of all impacts
```

**Оценка на вероятността от заплаха:**
- **Historical data** - анализ на честотата на минали инциденти
- **Industry statistics** - секторно специфично разузнаване на заплахи
- **Environmental factors** - оценка на текущия threat landscape
- **Control effectiveness** - оценка на съществуващите security мерки

**Пример за изчисляване на риска:**
```
E-commerce Customer Database:
├── Asset Value: $2,000,000
├── Threat: SQL Injection Attack
├── Vulnerability: Inadequate input validation
├── Probability: 30% annually (based on industry data)
├── Impact: 80% (partial data exposure)
└── Annual Risk = $2M × 0.30 × 0.80 = $480,000
```

**Качествена оценка на риска:**

**Подход с матрица на риска:**
```
Probability vs. Impact Matrix:
                Low Impact  Medium Impact  High Impact
High Probability    Medium      High       Critical
Med Probability     Low         Medium     High
Low Probability     Low         Low        Medium
```

**Стратегии за третиране на риска:**

** 1. Намаляване на риска (Reduce):**
- **Технически контроли** - firewalls, криптиране, контрол на достъпа
- **Административни контроли** - политики, процедури, обучение
- **Физически контроли** - заключалки, камери, контроли на средата
- **Съображения за разходите** - разходи за намаляване спрямо редукция на риска

** 2. Прехвърляне на риска (Share):**
- **Кибер застраховка** - прехвърляне на финансовия риск
- **Услуги на трети страни** - аутсорсване на рискови операции
- **Договорна отговорност** - споразумения за споделена отговорност
- **Облачни услуги** - прехвърляне на инфраструктурен риск

** 3. Приемане на риска (Accept):**
- **Рискове с ниско въздействие** - разходите за намаляване надхвърлят потенциалните загуби
- **Residual risk** - оставащ риск след намаляването
- **Одобрение от ръководството** - официална документация за приемане на риска
- **Редовен преглед** - периодична преоценка на риска

** 4. Избягване на риска (Eliminate):**
- **Премахване на функции** - елиминиране на рискова функционалност
- **Промяна на технологията** - преминаване към по-сигурни алтернативи

## **ЧАС 3: SECURITY REQUIREMENTS ENGINEERING И ARCHITECTURE DESIGN (40 мин)**

### **Разработване на Security изисквания (15 мин)**

**Категории на Security изисквания:**

**Изисквания за автентикация:**
```
Identity Verification Requirements:
├── Multi-factor authentication mandatory for admin access
├── Password complexity: 12+ characters, mixed case, numbers, symbols
├── Account lockout: 5 failed attempts, 15-minute lockout
├── Session timeout: 30 minutes inactivity for sensitive data
├── Password rotation: 90-day mandatory change for privileged accounts
└── Single sign-on integration with corporate identity provider
```

**Изисквания за авторизация:**
- **Role-based access control (RBAC)** - разрешения въз основа на служебни функции
- **Attribute-based access control (ABAC)** - динамични разрешения въз основа на контекст
- **Принцип на минималните привилегии** - минимално необходими разрешения
- **Разделение на задълженията** - критичните операции изискват множество одобрения

**Изисквания за защита на данните:**
```
Encryption Requirements:
├── Data at Rest: AES-256 encryption for all sensitive data
├── Data in Transit: TLS 1.3 for all network communications
├── Key Management: Hardware Security Module (HSM) for key storage
├── Database Encryption: Transparent Data Encryption (TDE)
└── Backup Encryption: Encrypted backups with separate key management

Data Classification Requirements:
├── Public: No special protection required
├── Internal: Access control and audit logging
├── Confidential: Encryption and need-to-know access
├── Restricted: Maximum security controls and monitoring
└── Data Loss Prevention (DLP) controls for all classifications
```

**Изисквания за одит и compliance:**
- **Изчерпателно логване** - всички security-релевантни събития
- **Целостност на логовете** - съхранение на логове, защитено от подправяне
- **Съхранение на логове** - периоди за съхранение, определени от compliance
- **Мониторинг в реално време** - незабавно алертиране при подозрителни дейности

**Техники за спецификация на security изисквания:**

**User Stories с security acceptance критерии:**

```
├── GIVEN user enters valid credentials
├── WHEN authentication is successful
├── THEN session is established with 30-minute timeout
├── AND login event is logged with timestamp и IP address
├── AND failed login attempts are tracked и limited
└── AND MFA is required for sensitive operations
```
