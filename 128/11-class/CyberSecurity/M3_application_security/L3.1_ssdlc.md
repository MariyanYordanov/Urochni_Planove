## 📚 **ЧАС 1: SSDLC PRINCIPLES И SECURITY-BY-DESIGN METHODOLOGY (40 мин)**

### **🏗️ Secure SDLC Overview и Framework (12 мин)**

**Traditional SDLC vs. Secure SDLC:**

```
Traditional SDLC Approach:
Requirements → Design → Development → Testing → Deployment
                                    ↑
                              Security Added Here
                              (Too Late, Expensive)

Secure SDLC Approach:
Security Requirements → Secure Design → Secure Development → Security Testing → Secure Deployment
    ↓                      ↓              ↓                    ↓                ↓
Security Planning → Threat Modeling → Secure Coding → Penetration Testing → Security Monitoring
```

**Security-by-Design Core Principles:**

**1. Proactive vs. Reactive Security:**
- **Shift-left approach** - integrate security от earliest phases
- **Cost of fixing** - 100x more expensive в production vs. design
- **Security debt** - accumulation на security issues over time
- **Prevention focus** - stop vulnerabilities before they're created

*Real-world example: Microsoft's SDL implementation reduced security vulnerabilities by 50%*

**2. Defense in Depth Strategy:**
- **Multiple security layers** - no single point of failure
- **Redundant protections** - backup security mechanisms
- **Layered validation** - input sanitization + WAF + database permissions
- **Gradual degradation** - system remains functional when layers fail

**3. Least Privilege Principle:**
- **Minimal access rights** - users/processes get only necessary permissions
- **Just-in-time access** - temporary elevated permissions when needed
- **Regular permission audits** - quarterly access reviews
- **Zero-trust architecture** - verify everything, trust nothing

**4. Fail-Safe Defaults:**
- **Secure by default** - applications start в secure state
- **Explicit permissions** - everything denied unless explicitly allowed
- **Error handling** - failures should not expose sensitive information
- **Configuration security** - secure defaults в all system settings

## 📚 **ЧАС 2: THREAT MODELING И RISK ASSESSMENT TECHNIQUES (40 мин)**

### **🎯 Threat Modeling Fundamentals (15 мин)**

**Threat Modeling Definition и Purpose:**

Threat modeling е systematic approach за identifying, cataloging, и prioritizing potential security threats към application или system. Целта е да се разберат attack vectors преди application deployment.

**Core Threat Modeling Questions:**
1. **What are we building?** - Asset identification и system understanding
2. **What can go wrong?** - Threat identification и attack scenario development
3. **What should we do about it?** - Mitigation strategy development
4. **Did we do a good job?** - Validation и continuous improvement

**STRIDE Threat Categorization:**

```
S - Spoofing Identity:
├── Impersonating legitimate users
├── Using stolen credentials
├── Session hijacking attacks
└── Mitigation: Strong authentication, MFA

T - Tampering with Data:
├── Unauthorized data modification
├── Man-in-the-middle attacks
├── Database injection attacks
└── Mitigation: Data integrity checks, encryption

R - Repudiation:
├── Denying performed actions
├── Log tampering attempts
├── Non-repudiation bypass
└── Mitigation: Audit logging, digital signatures

I - Information Disclosure:
├── Unauthorized data access
├── Data leakage scenarios
├── Privacy violations
└── Mitigation: Access controls, encryption

D - Denial of Service:
├── Resource exhaustion attacks
├── Service availability disruption
├── Performance degradation
└── Mitigation: Rate limiting, load balancing

E - Elevation of Privilege:
├── Unauthorized permission escalation
├── Administrative access bypass
├── Privilege abuse scenarios
└── Mitigation: Least privilege, role separation
```

**DREAD Risk Rating System:**

**Risk Calculation Formula:**
```
Risk Score = (Damage + Reproducibility + Exploitability + Affected Users + Discoverability) / 5

Rating Scale (1-10):
├── 1-3: Low Risk
├── 4-6: Medium Risk
├── 7-8: High Risk
└── 9-10: Critical Risk
```

**Example DREAD Assessment:**
```
SQL Injection Vulnerability:
├── Damage: 9 (Complete database access)
├── Reproducibility: 8 (Easy to reproduce)
├── Exploitability: 7 (Moderate skill required)
├── Affected Users: 9 (All users affected)
├── Discoverability: 6 (Moderate discovery difficulty)
└── Risk Score: 7.8 (High Risk)
```
