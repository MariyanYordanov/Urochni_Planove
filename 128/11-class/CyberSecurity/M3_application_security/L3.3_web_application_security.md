### **🔴 A01: Broken Access Control (10 мин)**

**Принцип и атаки:**

```
Access Control Failures:
┌─────────────────────────────────────────────────────────┐
│  IDOR (Insecure Direct Object Reference):               │
│                                                         │
│  Normal:  GET /api/orders/1001  → My order ✅           │
│  Attack:  GET /api/orders/1002  → Someone else's ❌     │
│                                                         │
│  Horizontal Privilege Escalation:                       │
│  User A accessing User B's data (same privilege level)  │
│                                                         │
│  Vertical Privilege Escalation:                         │
│  Regular user accessing admin functions                 │
│                                                         │
│  URL Manipulation:                                      │
│  /user/profile → /admin/users  (без проверка)          │
└─────────────────────────────────────────────────────────┘
```

**Mitigation:**

```python
# ОПАСНО — без ownership check:
@app.route('/api/document/<int:doc_id>')
def get_document(doc_id):
    return Document.get(doc_id)

# БЕЗОПАСНО — с authorization:
@app.route('/api/document/<int:doc_id>')
@login_required
def get_document(doc_id):
    doc = Document.get(doc_id)
    if doc is None:
        abort(404)
    if doc.owner_id != current_user.id and not current_user.is_admin:
        abort(403)  # Forbidden
    return doc.to_json()
```

**Принцип Deny by Default:**

```
Access Control Design:
┌─────────────────────────────────────────────────────────┐
│  ❌ ГРЕШНО: Allow all, deny exceptions                  │
│     Всичко достъпно ако не е изрично забранено          │
│                                                         │
│  ✅ ПРАВИЛНО: Deny all, allow exceptions                │
│     Нищо не е достъпно ако не е изрично разрешено       │
│                                                         │
│  Прилагай on EVERY request:                            │
│  1. Автентикация (кой е потребителят?)                 │
│  2. Авторизация (може ли да прави това?)               │
│  3. Ownership (негов ли е ресурсът?)                   │
└─────────────────────────────────────────────────────────┘
```

### **🔴 A03: Injection — SQL, XSS, Command (12 мин)**

**SQL Injection — механизъм:**

```
Normal Query:
SELECT * FROM users WHERE name = 'Alice'

SQL Injection Input: ' OR '1'='1
Resulting Query:
SELECT * FROM users WHERE name = '' OR '1'='1'
                                          ↑
                            Винаги вярно → всички записи!

Destructive SQLi: '; DROP TABLE users; --
SELECT * FROM users WHERE name = ''; DROP TABLE users; --'
```

**Защита срещу SQL Injection:**

```python
# ❌ ОПАСНО — String concatenation:
query = f"SELECT * FROM users WHERE email = '{email}'"
cursor.execute(query)

# ✅ БЕЗОПАСНО — Parameterized queries:
cursor.execute("SELECT * FROM users WHERE email = %s", (email,))

# ✅ ОЩЕ ПО-ДОБРЕ — ORM (SQLAlchemy):
user = User.query.filter_by(email=email).first()
# ORM автоматично параметризира всички queries
```

**Cross-Site Scripting (XSS) — типове:**

```
XSS Types:
┌─────────────────────────────────────────────────────────┐
│  REFLECTED XSS (Non-persistent):                        │
│  URL: /search?q=<script>document.cookie</script>        │
│  Server: <h1>Results for: [PAYLOAD]</h1>               │
│  Victim clicks malicious link → script изпълнен        │
│                                                         │
│  STORED XSS (Persistent):                               │
│  Attacker постава payload в comment/profile             │
│  Всеки, гледащ страницата → script изпълнен             │
│  Пример: коментар <script>стоук cookies</script>        │
│                                                         │
│  DOM-BASED XSS:                                         │
│  JavaScript четe от URL/DOM без sanitization            │
│  document.innerHTML = location.hash.substring(1)        │
└─────────────────────────────────────────────────────────┘
```

**XSS Mitigation:**

```javascript
// ❌ ОПАСНО:
element.innerHTML = userInput;
document.write(userInput);

// ✅ БЕЗОПАСНО:
element.textContent = userInput;  // Автоматично escaping

// За динамичен HTML — DOMPurify библиотека:
element.innerHTML = DOMPurify.sanitize(userInput);
```

**Content Security Policy (CSP) — допълнителна защита:**

```
HTTP Header:
Content-Security-Policy:
    default-src 'self';
    script-src 'self' https://trusted.cdn.com;
    style-src 'self' 'unsafe-inline';
    img-src *;
    frame-ancestors 'none';

Ефект: Дори при XSS, inline scripts са блокирани!
```

## 📚 **ЧАС 2: INPUT VALIDATION И OUTPUT ENCODING (40 мин)**

### **🔒 Принципи на input validation (12 мин)**

**Защо validation е критична:**

```
Input Validation Philosophy:
┌─────────────────────────────────────────────────────────┐
│  "Всеки потребителски вход е потенциално злонамерен"    │
│                                                         │
│  Validation трябва да е:                                │
│  1. SERVER-SIDE (не само client-side!)                  │
│     └── JavaScript validation лесно се bypass-ва        │
│                                                         │
│  2. WHITELIST (а не само blacklist)                     │
│     ❌ Blacklist: Забрани "<script>", "DROP", etc.     │
│        Проблем: Атакуващите намират обходи              │
│     ✅ Whitelist: Позволи само очаквания формат         │
│        Email: /^[a-zA-Z0-9._%+-]+@[a-zA-Z0-9.-]+\.\w+$/│
│                                                         │
│  3. FAIL FAST — откажи при невалидни данни              │
│     └── Не се опитвай да "почистиш" входа              │
└─────────────────────────────────────────────────────────┘
```

**Validation примери:**

```python
from pydantic import BaseModel, EmailStr, validator
import re

class UserRegistration(BaseModel):
    username: str
    email: EmailStr
    age: int

    @validator('username')
    def username_alphanumeric(cls, v):
        if not re.match(r'^[a-zA-Z0-9_]{3,20}$', v):
            raise ValueError('Username: само букви, цифри, _ (3-20 символа)')
        return v

    @validator('age')
    def valid_age(cls, v):
        if not (13 <= v <= 120):
            raise ValueError('Невалидна възраст')
        return v
```

## 📚 **ЧАС 3: AUTHENTICATION И SESSION MANAGEMENT SECURITY (40 мин)**

### **🔐 Secure Password Management (15 мин)**

**Password Hashing — правилен подход:**

```
Password Storage Evolution:
┌─────────────────────────────────────────────────────────┐
│  ❌ Plaintext: "password123"  (директно в БД!)          │
│                                                         │
│  ❌ MD5: 482c811da5d5b4bc6d497ffa98491e38              │
│     Cracked в секунди с rainbow tables                  │
│                                                         │
│  ❌ SHA-256 (без salt):                                 │
│     Различни потребители, същата парола → Същият hash  │
│     → Разкрива кой ползва същата парола!               │
│                                                         │
│  ✅ bcrypt (cost factor 12):                            │
│     $2b$12$K9L7Y... (автоматично salt + iterations)    │
│     ~300ms за verify (прави brute force непрактичен)    │
│                                                         │
│  ✅ Argon2id (OWASP препоръчан, 2023):                 │
│     Memory-hard: изисква RAM → GPU cracking труден      │
│     Параметри: m=64MB, t=3, p=4                        │
└─────────────────────────────────────────────────────────┘
```

**Имплементация с bcrypt:**

```python
import bcrypt

# При регистрация:
def register_user(username, plaintext_password):
    # Генерира salt автоматично, cost=12 (препоръчан)
    hashed = bcrypt.hashpw(plaintext_password.encode(),
                           bcrypt.gensalt(rounds=12))
    db.save_user(username, hashed)

# При login:
def login_user(username, plaintext_password):
    user = db.get_user(username)
    if user and bcrypt.checkpw(plaintext_password.encode(),
                                user.password_hash):
        return create_session(user)
    else:
        # Важно: Един и същ error message!
        raise AuthError("Невалидно потребителско име или парола")
```


### **🔑 Multi-Factor Authentication (MFA) (12 мин)**

**MFA фактори:**

```
Authentication Factors:
┌─────────────────────────────────────────────────────────┐
│  Something you KNOW (Знание):                           │
│  ├── Парола                                             │
│  ├── PIN                                               │
│  └── Security questions (слаб метод!)                  │
│                                                         │
│  Something you HAVE (Притежание):                       │
│  ├── TOTP (Google Authenticator, Authy) — препоръчан   │

│  ├── Hardware key (YubiKey, FIDO2) — най-силен         │
│  └── Email OTP                                         │
│                                                         │
│  Something you ARE (Биометрия):                         │
│  ├── Fingerprint                                       │
│  ├── Face recognition                                  │
│  └── Voice print                                       │
│                                                         │
│  MFA изисква ПОНЕ 2 различни фактора                   │
└─────────────────────────────────────────────────────────┘
```

**TOTP Implementation (RFC 6238):**

```python
import pyotp

# При активиране на MFA:
def setup_mfa(user):
    secret = pyotp.random_base32()  # Генерира secret
    user.mfa_secret = secret        # Запазва в БД (encrypted!)

    # QR code URI за Google Authenticator:
    totp_uri = pyotp.totp.TOTP(secret).provisioning_uri(
        name=user.email,
        issuer_name="MyApp"
    )
    return totp_uri  # Покажи като QR code на потребителя

# При верификация:
def verify_mfa(user, code):
    totp = pyotp.TOTP(user.mfa_secret)
    return totp.verify(code, valid_window=1)
    # valid_window=1 позволява ±30 сек clock skew
```

```python
    'editor':  ['read', 'write'],
    'admin':   ['read', 'write', 'delete', 'manage_users'],
}

def require_permission(permission):
    """Декоратор за permission-based access control"""
    def decorator(f):
        @wraps(f)
        def decorated_function(*args, **kwargs):
            if not current_user.is_authenticated:
                abort(401)  # Unauthorized
            user_permissions = ROLES_PERMISSIONS.get(
                current_user.role, []
            )
            if permission not in user_permissions:
                abort(403)  # Forbidden
            return f(*args, **kwargs)
        return decorated_function
    return decorator

# Употреба:
@app.route('/admin/users')
@require_permission('manage_users')
def manage_users():
    return render_template('admin/users.html')

@app.route('/document/<int:id>', methods=['DELETE'])
@require_permission('delete')
def delete_document(id):
    doc = Document.get_or_404(id)
    # Допълнителна проверка за ownership!
    if doc.owner_id != current_user.id and \
       'manage_all' not in current_user.permissions:
        abort(403)
    doc.delete()
    return '', 204
```

### **🔍 CSRF (Cross-Site Request Forgery) (13 мин)**

**Как работи CSRF атаката:**

```
CSRF Attack Flow:
┌─────────────────────────────────────────────────────────┐
│  1. Жертвата е логнат в bank.com (има session cookie)  │
│                                                         │
│  2. Жертвата посещава evil.com                         │
│                                                         │
│  3. evil.com съдържа скрита форма:                     │
│     <form action="https://bank.com/transfer" method="POST">│
│       <input type="hidden" name="amount" value="1000"> │
│       <input type="hidden" name="to" value="attacker"> │
│     </form>                                             │
│     <script>document.forms[0].submit();</script>       │
│                                                         │
│  4. Browser автоматично изпраща session cookie!        │
│     → Transfer изпълнен без знанието на жертвата!       │
└─────────────────────────────────────────────────────────┘
```

**CSRF Mitigation:**

```python
# Flask-WTF автоматично CSRF protection:
from flask_wtf import FlaskForm
from wtforms import StringField

class TransferForm(FlaskForm):
    amount = StringField('Amount')
    recipient = StringField('Recipient')
    # Flask-WTF автоматично добавя hidden csrf_token field

# Template:
# <form method="POST">
#     {{ form.hidden_tag() }}  ← CSRF token
#     {{ form.amount() }}
#     <button type="submit">Transfer</button>
# </form>

# Verification е автоматично при form validation:
@app.route('/transfer', methods=['POST'])
def transfer():
    form = TransferForm()
    if form.validate_on_submit():  # Validates CSRF token!
        process_transfer(form.amount.data, form.recipient.data)
```

**SameSite Cookie — допълнителна CSRF защита:**

```
SameSite Values:
├── Strict: Cookie НЕ се изпраща при cross-site requests
│           (максимална защита, може да счупи OAuth flows)
├── Lax: Cookie се изпраща само при GET navigation
│        (балансиран, препоръчан default)
└── None: Cookie се изпраща при всякакви requests
         (изисква Secure; нужен за cross-site OAuth)
```

**Manual Testing — OWASP тестове:**

```
Тест за SQL Injection:
1. Намери input field (search, login, ID parameter)
2. Въведи: '  (single quote)
3. Ако SQL грешка → потенциален SQLi!
4. Тествай: ' OR '1'='1
5. Тествай: ' OR 1=1 --
6. Използвай sqlmap САМО с разрешение

Тест за XSS:
1. Намери reflected input (search, error messages)
2. Въведи: <script>alert('XSS')</script>
3. Ако alert се изпълни → XSS уязвимост!
4. Тествай в различни контексти: HTML, attributes, JS
5. Опитай: <img src=x onerror=alert(1)>

Тест за IDOR:
1. Намери numeric ID в URL: /api/order/1001
2. Промени на: /api/order/1002
3. Ако получиш чужди данни → IDOR!
4. Тествай DELETE/PUT методи също

Тест за Authentication:
1. Опитай default credentials (admin/admin)
2. Тествай password reset flow (token expiry, reuse)
3. Провери brute force protection (lockout?)
4. Тествай session invalidation при logout
```
