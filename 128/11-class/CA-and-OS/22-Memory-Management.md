### **3. НОВ МАТЕРИАЛ (42 минути)**

#### **Част 1: Защо е нужно управление на памет? (8 мин)**

**Проблеми за решаване:**
```
1. Ограничен размер на RAM
   - RAM: 16 GB
   - OS: 2-4 GB
   - Браузър: 2-8 GB
   - Игра: 8-16 GB
   - Други: 2-4 GB
   = Често > 16 GB общо!

2. Множество процеси едновременно
   - Изолация между процеси
   - Защита на паметта
   - Споделяне (libraries)

3. Ефективност
   - Минимизиране на фрагментация
   - Бърз достъп
```

**Йерархия на паметта:**
```
Скорост ▲         │ Капацитет
        │ Регистри (CPU)     ~KB      < 1 ns
        │ L1 Cache           32-64 KB  ~1 ns
        │ L2 Cache           256 KB-1MB ~4 ns
        │ L3 Cache           8-64 MB   ~10 ns
        │ RAM                8-128 GB  ~100 ns
        │ SSD (NVMe)         256GB-4TB ~100 μs
        ▼ HDD                1-20 TB   ~10 ms
```

#### **Част 2: Логически vs Физически адреси (8 мин)**

```
ЛОГИЧЕСКИ (ВИРТУАЛНИ) АДРЕСИ:
├── Генерирани от CPU
├── Виждани от програмата
├── Започват от 0 за всеки процес
└── Независими от физическата памет

ФИЗИЧЕСКИ АДРЕСИ:
├── Реални адреси в RAM
├── Използвани от Memory Controller
├── Уникални за цялата система
└── Определят реалната локация

         Процес                   RAM
    ┌────────────┐           ┌────────────┐
    │  0x0000    │           │  0x10000   │ P2
    │  0x1000    │──────────►│  0x20000   │ P1
    │  0x2000    │    MMU    │  0x30000   │ P3
    │  ...       │  (Memory  │  0x40000   │ P1
    └────────────┘   Mgmt    │  ...       │
                     Unit)   └────────────┘

MMU преобразува логически → физически адреси
```

#### **Част 3: Contiguous Allocation (6 мин)**

```
Всеки процес получава непрекъснат блок памет.

RAM:
┌────────────────────────────────────────────┐
│ OS │   P1   │ Свободно │   P2   │  P3  │   │
└────────────────────────────────────────────┘

Проблеми:
1. External Fragmentation
   - Има свободна памет, но не е непрекъсната
   
   │ OS │ P1 │ Free │ P2 │ Free │ P3 │ Free │
   
   Общо свободно: 300 MB
   Нужно за P4: 250 MB
   Но най-голямото парче: 150 MB
   → Не може да се зареди P4!

Решение: Compaction (преместване) - скъпо!
```

#### **Част 4: Paging - Странициране (12 мин)**

**Концепция:**
```
Раздели паметта на фиксирани блокове:
- Page (страница): блок в логическата памет
- Frame (рамка): блок във физическата памет
- Типичен размер: 4 KB

Предимства:
✓ Няма external fragmentation
✓ Не е нужна непрекъсната физическа памет
✓ Лесно управление
```

**Пример:**
```
Логическа памет (P1):       Физическа памет (RAM):
┌────────┐ Page 0           ┌────────┐ Frame 0 - P2
├────────┤ Page 1           ├────────┤ Frame 1 - P1 Page 0
├────────┤ Page 2           ├────────┤ Frame 2 - P3
├────────┤ Page 3           ├────────┤ Frame 3 - P1 Page 2
└────────┘                  ├────────┤ Frame 4 - Free
                            ├────────┤ Frame 5 - P1 Page 1
                            ├────────┤ Frame 6 - P1 Page 3
                            └────────┘

Page Table за P1:
┌──────────┬──────────┐
│ Page #   │ Frame #  │
├──────────┼──────────┤
│    0     │    1     │
│    1     │    5     │
│    2     │    3     │
│    3     │    6     │
└──────────┴──────────┘
```

**Превод на адрес:**
```
Логически адрес: Page Number + Offset

Ако размер на страница = 4 KB = 4096 bytes = 2^12
Адрес = 32 bits

Адрес: 0x00005A3C
       ├────────┤├──────┤
       Page #    Offset
       (20 bits) (12 bits)

Page # = 0x00005 = 5
Offset = 0xA3C = 2620

Page Table[5] = Frame 12 (например)
Физически адрес = Frame 12 × 4096 + 2620
                = 49152 + 2620 = 51772
                = 0x0000CA3C
```

#### **Част 5: Виртуална памет (8 мин)**

**Концепция:**
```
Използвай диска като "разширение" на RAM.

Виртуална памет > Физическа RAM

┌─────────────────┐
│ Виртуална памет │  (напр. 8 GB за процес)
│   за процес     │
└────────┬────────┘
         │
         ▼
┌─────────────────┐   ┌─────────────────┐
│  RAM (16 GB)    │   │  Swap/Page File │
│  Активни pages  │ ◄─► (на SSD/HDD)   │
└─────────────────┘   └─────────────────┘
```

**Page Fault:**
```
1. Програмата иска достъп до адрес
2. MMU проверява Page Table
3. Ако Valid bit = 0 (страницата не е в RAM):
   → PAGE FAULT!
   
4. OS прекъсва процеса
5. Зарежда страницата от диска в RAM
6. Обновява Page Table
7. Рестартира инструкцията

Page fault е СКЪПА операция! (~10 ms vs ~100 ns)
```

**Swap Space:**
```
Windows: pagefile.sys (Page File)
Linux: swap partition или swap file

Размер: Обикновено 1-2× RAM
- 16 GB RAM → 16-32 GB swap

Наблюдение в Task Manager:
- Committed memory > Physical memory
- Page Faults/sec
```

### **5. ОБОБЩЕНИЕ (8 минути)**

**Ключови точки:**
```
1. MMU превежда логически → физически адреси
2. Paging разделя паметта на страници с фиксиран размер
3. Page Table пази съответствието page → frame
4. Виртуалната памет използва диска като разширение
5. Page fault = страницата не е в RAM → зареди от диска
6. Твърде много page faults = thrashing = бавна система
```

## 7. ПРИЛОЖЕНИЯ

### ПРИЛОЖЕНИЕ 1: ФОРМУЛИ

```
Размер на страница = 2^n bytes (n = offset bits)
Page Number = Address >> n
Offset = Address & (2^n - 1)
Physical Address = Frame × Page_Size + Offset
```
