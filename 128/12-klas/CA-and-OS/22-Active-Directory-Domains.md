### **3. НОВ МАТЕРИАЛ (45 минути)**

#### **Част 1: Какво е Active Directory? (10 минути)**

**Дефиниция:**
```
ACTIVE DIRECTORY (AD):
├── Directory Service на Microsoft
├── Централизирана база данни за обекти в мрежата
├── Въведена с Windows 2000
├── Използва LDAP (Lightweight Directory Access Protocol)
└── Интегрирана с DNS

Какво съхранява AD:
├── Потребители и техните атрибути
├── Групи и членство
├── Компютри и сървъри
├── Политики за сигурност
├── Приложения и услуги
└── Споделени ресурси (принтери, папки)
```

**Предимства на AD:**
```
┌─────────────────────────────────────────────────────────────┐
│                  ПРЕДИМСТВА НА AD                           │
├─────────────────────────────────────────────────────────────┤
│ 1. ЦЕНТРАЛИЗИРАНО УПРАВЛЕНИЕ                                │
│    • Един акаунт = достъп навсякъде                        │
│    • Промяна на парола от едно място                       │
│    • Групови политики за всички машини                     │
├─────────────────────────────────────────────────────────────┤
│ 2. SINGLE SIGN-ON (SSO)                                     │
│    • Едно влизане = достъп до всички ресурси               │
│    • Kerberos автентикация                                  │
├─────────────────────────────────────────────────────────────┤
│ 3. СИГУРНОСТ                                                │
│    • Централен контрол на достъпа                          │
│    • Одит на действията                                     │
│    • Политики за пароли                                     │
├─────────────────────────────────────────────────────────────┤
│ 4. МАЩАБИРУЕМОСТ                                            │
│    • От 10 до милиони обекти                               │
│    • Репликация за надеждност                              │
└─────────────────────────────────────────────────────────────┘
```

#### **Част 2: AD структура - Домейни, Дървета, Гори (15 минути)**

**Йерархия на AD:**
```
                    ┌──────────────────────┐
                    │      FOREST          │
                    │  (company.com)       │
                    └──────────┬───────────┘
                               │
           ┌───────────────────┴───────────────────┐
           │                                       │
    ┌──────┴──────┐                        ┌──────┴──────┐
    │    TREE     │                        │    TREE     │
    │ europe.     │                        │ asia.       │
    │ company.com │                        │ company.com │
    └──────┬──────┘                        └──────┬──────┘
           │                                       │
    ┌──────┴──────┐                        ┌──────┴──────┐
    │   DOMAIN    │                        │   DOMAIN    │
    │  bg.europe. │                        │ jp.asia.    │
    │ company.com │                        │ company.com │
    └─────────────┘                        └─────────────┘

Терминология:
├── FOREST (Гора): Най-високо ниво, съдържа всички домейни
├── TREE (Дърво): Йерархия от домейни със споделен namespace
├── DOMAIN (Домейн): Основна административна единица
└── Всички домейни в една гора споделят Global Catalog
```

**Домейн в детайли:**
```
ДОМЕЙН:
├── Административна граница
├── Граница за репликация
├── Политики за сигурност
├── Съдържа:
│   ├── Domain Controller(s)
│   ├── Organizational Units
│   ├── Потребители
│   ├── Групи
│   └── Компютри
│
└── Пример: school.local
```

**Domain Controller (DC):**
```
DOMAIN CONTROLLER:
├── Сървър с AD роля
├── Съхранява копие на AD базата данни
├── Обработва автентикация
├── Репликира промени с други DC
│
├── NTDS.dit - базата данни на AD
├── SYSVOL - споделена папка за политики
│
└── Best Practice: Минимум 2 DC за redundancy
```

**FSMO роли:**
```
FSMO (Flexible Single Master Operations):
├── Schema Master - една за гората
├── Domain Naming Master - една за гората
├── RID Master - една за домейн
├── PDC Emulator - една за домейн
└── Infrastructure Master - една за домейн

Тези роли определят кой DC отговаря за 
конкретни операции в AD.
```

#### **Част 3: Organizational Units (OU) (10 минути)**

**OU концепция:**
```
ORGANIZATIONAL UNIT (OU):
├── Контейнер за организиране на обекти
├── Позволява делегиране на права
├── Може да прилага Group Policy
├── Йерархична структура
│
└── Разлика от ГРУПИ:
    ├── OU е КОНТЕЙНЕР (съдържа обекти)
    └── Група е КОЛЕКЦИЯ (потребители за права)

OU ДИЗАЙН ПОДХОДИ:
├── По департамент (HR, IT, Sales)
├── По локация (Sofia, Plovdiv)
├── По функция (Servers, Workstations, Users)
└── Комбинация от горните
```

**Примерна OU структура:**
```
school.local (Domain)
│
├── OU=Students
│   ├── OU=Grade11
│   └── OU=Grade12
│
├── OU=Teachers
│   ├── OU=IT_Department
│   └── OU=Math_Department
│
├── OU=Computers
│   ├── OU=ClassroomPCs
│   └── OU=TeacherPCs
│
├── OU=Servers
│
└── OU=ServiceAccounts
```

**Делегиране на права:**
```
OU делегиране позволява:
├── IT администратор да управлява OU=Computers
├── HR да ресетва пароли в OU=Employees
├── Department lead да създава потребители в своя OU
│
└── Принцип на минималните права:
    Всеки има достъп само до това, което му трябва
```

#### **Част 4: AD обекти (10 минути)**

**Основни обекти:**
```
USER (Потребител):
├── Уникален идентификатор в AD
├── Атрибути:
│   ├── sAMAccountName (login name)
│   ├── userPrincipalName (UPN: user@domain.com)
│   ├── displayName
│   ├── mail
│   ├── telephoneNumber
│   └── department, title, manager...
│
├── Distinguished Name (DN):
│   CN=John Smith,OU=Sales,DC=company,DC=com
│
└── Пример: john.smith@school.local

GROUP (Група):
├── Колекция от потребители/компютри/други групи
├── Типове:
│   ├── Security Group - за права
│   └── Distribution Group - за email
├── Scope:
│   ├── Domain Local - права в домейна
│   ├── Global - потребители от домейна
│   └── Universal - цялата гора
│
└── Best Practice: AGDLP
    Account → Global → Domain Local → Permission

COMPUTER:
├── Компютър в домейна
├── Има машинен акаунт (COMPUTER$)
├── Може да получава Group Policy
└── Trust relationship с домейна
```

**Примерна структура:**
```
OU=IT_Department
├── User: admin.ivan
├── User: support.maria
├── Group: IT_Admins
│   └── Members: admin.ivan
├── Group: IT_Support
│   └── Members: support.maria, IT_Admins
│
└── Computer: PC-IT-001
```

### **4. ДЕМОНСТРАЦИЯ / ПРАКТИКА (20 минути)**

**Ако има Windows Server VM:**
```powershell
# PowerShell AD команди

# Преглед на домейна
Get-ADDomain

# Списък на OU
Get-ADOrganizationalUnit -Filter *

# Потребители
Get-ADUser -Filter * | Select-Object Name, SamAccountName

# Групи
Get-ADGroup -Filter * | Select-Object Name, GroupCategory, GroupScope

# Компютри
Get-ADComputer -Filter * | Select-Object Name, OperatingSystem

# Създаване на OU
New-ADOrganizationalUnit -Name "TestOU" -Path "DC=school,DC=local"

# Създаване на потребител
New-ADUser -Name "Test User" `
  -SamAccountName "testuser" `
  -UserPrincipalName "testuser@school.local" `
  -Path "OU=TestOU,DC=school,DC=local" `
  -AccountPassword (ConvertTo-SecureString "P@ssw0rd!" -AsPlainText -Force) `
  -Enabled $true

# Създаване на група
New-ADGroup -Name "TestGroup" `
  -GroupScope Global `
  -GroupCategory Security `
  -Path "OU=TestOU,DC=school,DC=local"

# Добавяне на член към група
Add-ADGroupMember -Identity "TestGroup" -Members "testuser"
```

**Ако няма Windows Server - симулация:**
```
Упражнение: Проектирайте AD структура за:

КОМПАНИЯ: TechCorp Ltd.
├── 3 офиса: София, Пловдив, Варна
├── 500 служители
├── Отдели: IT, HR, Finance, Sales, Marketing
├── Типове компютри: Desktop, Laptop, Servers

Задачи:
1. Начертайте OU структура
2. Определете какви групи са нужни
3. Покажете как потребител получава достъп до споделена папка
```
