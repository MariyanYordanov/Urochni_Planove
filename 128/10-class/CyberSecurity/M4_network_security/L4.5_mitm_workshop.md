## Учебно съдържание (40 минути)

### 1. Какво е MITM атака (8 мин)

**Man-in-the-Middle (MITM)** е вид атака, при която нападателят се позиционира тайно между двама комуникиращи участници, прихваща и евентуално модифицира техния трафик, без те да знаят.

**Диаграма на атаката:**

```
БЕЗ АТАКА:
  Жертва ──────────────────────── Сървър
  (изпраща данни директно)

С MITM АТАКА:
  Жертва ──── Нападател ──── Сървър
              (прихваща,
               чете,
               препраща)
```

**Какво може нападателят да направи:**
- Да **чете** некриптирания трафик (пароли, съобщения, cookie-та)
- Да **модифицира** съдържанието на комуникацията
- Да **инжектира** зловреден код или фалшиви данни
- Да **записва** целия трафик за по-късен анализ

**Предпоставки за успешна MITM атака:**
- Нападателят е в същата мрежа като жертвата (или контролира мрежова точка по пътя)
- Комуникацията е некриптирана или криптирането е компрометирано

### 2. Видове MITM атаки (12 мин)

#### ARP Spoofing (ARP Poisoning)

**Как работи:**
- ARP (Address Resolution Protocol) свързва IP адреси с MAC адреси в локална мрежа
- Нападателят изпраща фалшиви ARP отговори: „IP на gateway-а = моят MAC адрес"
- Жертвата обновява ARP таблицата си и насочва целия трафик към нападателя
- Нападателят препраща трафика към истинския gateway (victim не забелязва)

**Индикатор в Wireshark:** Duplicate IP address detected — два различни MAC адреса за един IP

```
ARP Reply: 192.168.1.1 is at AA:BB:CC:DD:EE:FF  ← легитимен gateway
ARP Reply: 192.168.1.1 is at 11:22:33:44:55:66  ← нападател (дублиран IP!)
```

#### DNS Spoofing (DNS Cache Poisoning)

**Как работи:**
- Нападателят манипулира DNS отговорите
- Когато жертвата иска IP адреса на `bank.bg`, получава фалшив IP
- Браузърът отваря фалшив сайт, идентичен на истинския (phishing)
- Жертвата въвежда данни на фалшивия сайт

**Индикатор:** DNS отговор с необичаен IP адрес за известен домейн

#### SSL Stripping

**Как работи:**
- Жертвата иска да отвори `https://bank.bg`
- Нападателят (между жертвата и сървъра) поддържа HTTPS връзка към сървъра
- Но към жертвата отговаря с HTTP (некриптирана) версия
- Жертвата вижда сайта, но без HTTPS — данните вървят некриптирани

**Индикатор:** Браузърът показва HTTP вместо HTTPS за сайт, който трябва да е HTTPS

#### Evil Twin Wi-Fi

**Как работи:**
- Нападателят създава фалшива Wi-Fi точка с идентично или подобно SSID на легитимна мрежа
- Жертвата се свързва към фалшивата мрежа (по-силен сигнал или deauth атака)
- Целият трафик минава през нападателя

**Пример:** В кафене с мрежа „Cafe_Free_WiFi" нападателят пуска „Cafe_Free_WiFi" или „CafeFreeWiFi"

### 3. Реални примери (5 мин)

**Публични Wi-Fi мрежи:**
- Летища, хотели, кафенета — честа среда за Evil Twin и ARP spoofing атаки
- Потребители, проверяващи банкови сметки на публичен Wi-Fi, са уязвими
- Проучване на Kaspersky (2019): 25% от Wi-Fi hotspot-овете са незащитени

**Корпоративни атаки:**
- Вътрешен нападател в корпоративна мрежа може да прихваща вътрешна комуникация
- ARP spoofing срещу VoIP телефони — подслушване на разговори
- Атаки срещу IoT устройства в корпоративна мрежа (камери, принтери)

**Исторически случаи:**
- Superfish adware (Lenovo, 2015) — инсталираше собствен SSL certificate, позволяваше SSL stripping
- DigiNotar (2011) — компрометиран Certificate Authority, издаден фалшив сертификат за Google

### 4. Защита срещу MITM атаки (10 мин)

#### HTTPS и TLS

- Винаги използвай HTTPS — TLS криптира трафика, прави го нечетим за нападателя
- Провери дали URL-ът започва с `https://` и има валиден сертификат (катинарче)
- Никога не приемай предупреждения за невалиден сертификат без проверка

#### Certificate Pinning

- Приложенията (особено мобилни) „запомнят" конкретния сертификат на сървъра
- Ако сертификатът се смени (напр. от MITM proxy), приложението отказва връзката
- Използва се в банкови приложения и критични системи

#### VPN (Virtual Private Network)

- VPN криптира целия трафик между устройството и VPN сървъра
- Дори в компрометирана мрежа, нападателят вижда само криптиран трафик
- Важно: изборът на доверен VPN доставчик е критичен

#### 802.1X Network Authentication

- Корпоративен стандарт за удостоверяване при свързване към мрежа
- Всяко устройство трябва да се удостовери преди да получи достъп
- Предотвратява свързване на неоторизирани устройства (Evil Twin е по-трудна)

#### Допълнителни мерки

| Мярка | Защита срещу |
|-------|-------------|
| Dynamic ARP Inspection (DAI) на switch-а | ARP Spoofing |
| DNSSEC | DNS Spoofing |
| HTTP Strict Transport Security (HSTS) | SSL Stripping |
| Wi-Fi мрежа с WPA3 | Evil Twin (частично) |
| Мониторинг на ARP таблицата | ARP Spoofing |

### 5. Демонстрация с Wireshark: ARP spoofing в packet capture (5 мин)

**Учителят демонстрира на проектора:**

1. Отвори Wireshark с предоставения тестов .pcap файл
2. Приложи filter: `arp`
3. Покажи нормалните ARP заявки и отговори
4. Идентифицирай момента на атаката:
   - Рязко увеличаване на ARP пакети (ARP storm)
   - Два различни MAC адреса, отговарящи за един IP
5. Разгъни пакета и покажи:
   - Sender MAC address (нападателят)
   - Sender IP address (IP на gateway-а — фалшиво)

**Wireshark маркира проблема автоматично:**
- Пакети с предупреждение „Duplicate IP address detected" се оцветяват
- Info колоната показва „ARP Reply" с подозрителни стойности

## Практическо упражнение: Анализ на pcap файл с ARP spoofing (включен в часа)

**Задача:** Анализирай предоставения .pcap файл и отговори на въпросите.

**Файл:** `resources/Wireshark/arp_spoofing_sample.pcap` (предоставен от учителя)

**Стъпки:**
1. Отвори файла с Wireshark: File → Open
2. Приложи filter: `arp`
3. Разгледай ARP пакетите хронологично

**Въпроси за анализ:**
1. Кой е IP адресът на gateway-а в мрежата?
2. Кои два MAC адреса претендират за един и същи IP? Запиши ги.
3. В кой момент (времеви timestamp) започва атаката?
4. Как се казва предупреждението, което Wireshark показва?
5. Какъв е MAC адресът на вероятния нападател?

**Бонус въпрос:** Свали filter-а и разгледай HTTP трафика — можеш ли да намериш данни, прихванати по време на атаката?
